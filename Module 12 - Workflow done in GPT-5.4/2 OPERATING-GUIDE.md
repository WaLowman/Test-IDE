# Johannesburg GTM Workflow Package

This package is a lightweight, Markdown-first sub-agent workflow for a Johannesburg tour and transport business selling curated cultural experiences to business, diplomatic, and international visitors in Gauteng.

It is designed for fast demo use, not full operational completeness.

## What This Workflow Does

The workflow moves through three commercial pillars:

- `clients`: organisations and individuals who may buy experiences directly.
- `partners`: intermediaries who can refer, bundle, host, or resell.
- `experiences`: curated products and provider combinations worth packaging and selling.

Each pillar runs through four stages:

1. `discovery`
2. `ranking`
3. `research`
4. `outreach`

The stages are sequential. Each stage must leave clean outputs for the next stage.

## Core Assumptions

- Geography is Johannesburg-first, Gauteng-wide.
- Buyers are business visitors, diplomatic missions, trade delegations, executive guests, conference organisers, relocation support teams, and international travelers with structured itineraries.
- The offer combines transport, cultural access, hosting, and concierge-style reliability.
- This package favours clarity, repeatability, and speed over exhaustive market coverage.

## Package Layout

- `shared/`: operating assumptions, templates, and scoring rules.
- `agents/`: one brief per agent.
- `runs/`: dated execution folders generated by the runner.
- `run-workflow.sh`: one-command scaffold for a fresh dated run.
- `RUN.md`: quick start.

## Operating Rules

- All outputs are dated using `YYYY-MM-DD`.
- All entity slugs use lowercase kebab-case.
- Every stage writes into the current run folder, never into the agent brief folders.
- Discovery finds candidates.
- Ranking reduces the list and assigns priority.
- Research builds usable dossiers only on shortlisted names.
- Outreach drafts specific messages only from research-backed dossiers.

## Standard Run Shape

Each run folder uses this structure:

```text
runs/YYYY-MM-DD-jhb-gauteng-gtm/
  clients/
    01-discovery/
    02-ranking/
    03-research/
    04-outreach/
  partners/
    01-discovery/
    02-ranking/
    03-research/
    04-outreach/
  experiences/
    01-discovery/
    02-ranking/
    03-research/
    04-outreach/
  run-index.md
```

## Minimum Execution Standard

For a clean demo:

- Discovery should identify at least `15` candidates per pillar.
- Ranking should shortlist the top `5`.
- Research should produce dossiers on the top `3`.
- Outreach should draft at least `2` tailored messages and one call-prep note per pillar.

## Handoff Discipline

Do not skip gates:

- No ranking without a discovery table.
- No research without a ranked shortlist.
- No outreach without a dossier.

If a stage is blocked, document the blocker in the stage folder and stop the branch cleanly.

## Suggested Demo Flow

1. Run `./run-workflow.sh`
2. Open the new `runs/<date>-jhb-gauteng-gtm/run-index.md`
3. Work stage by stage using the matching agent briefs in `agents/`
4. Keep notes short, sourced, and decision-oriented
