# AGENTIC GATEWAYS

**How OpenClaw Turned Messaging Apps into Remote Control Surfaces for Personal Agents**

## Next Up

Claude Code showed that an agent could work on your machine.

Claude Cowork showed that the same architecture could be packaged as a desktop workbench for broader knowledge work.

Agentic gateways take the next step. They ask a different question:

**What if your agent does not live mainly in a terminal or desktop app, but is reachable through the messaging channels you already use?**

That is the conceptual role of an **Agentic Gateway**. It is not the agent itself. It is the control plane and communications layer that sits between:

- Chat surfaces such as WhatsApp, Telegram, Discord, iMessage, or Slack.
- The model provider.
- The local or hosted runtime.
- The tools, plugins, files, and credentials the agent can use.

In the taxonomy of this course, that is why OpenClaw sits under **Agentic Gateways** rather than under workbenches or coding tools.

## What OpenClaw Is

OpenClaw's own documentation describes it as a self-hosted gateway that connects messaging apps to AI agents. Its docs present the Gateway as the central WebSocket control plane for channels, sessions, nodes, hooks, and agent calls. Its homepage makes the product vision even clearer: message the agent from your pocket, but let it operate against a machine, model stack, and tool environment you control.

That mattered because it changed the surface area of agent use.

Instead of opening an IDE or desktop app, the user could:

- Send a message from a phone.
- Trigger a multi-step task remotely.
- Let the agent operate through a persistent gateway.
- Receive results back in a familiar messaging thread.

This is why OpenClaw was such an important moment. It reframed the agent as something closer to a **personal operating layer** than a chatbot.

## Why OpenClaw Took Off

OpenClaw did not become interesting because it had the best model. It became interesting because it solved a product problem that many labs had only partially addressed:

- Users wanted their agent reachable from anywhere.
- Users wanted the agent to persist across sessions.
- Users wanted the agent to touch real systems and real files.
- Users wanted one control surface that sat above many tools and channels.

OpenClaw's model was simple and powerful:

1. Run the gateway once.
2. Connect your channels and tools.
3. Message your agent from wherever you already are.

That is a very different value proposition from "open a chat window and ask a question."

## Why OpenClaw Caused So Much Concern

The same design that made OpenClaw exciting also made it dangerous.

A self-hosted gateway with durable credentials, remote messaging access, plugin installation, memory, and local execution creates a much larger attack surface than a conventional chatbot. OpenClaw's own security docs acknowledge the need for audits, trust-boundary separation, safer DM scoping, auth hardening, and warnings around dangerous flags. Microsoft went further in an official February 19, 2026 security blog post, arguing that OpenClaw should be treated as **untrusted code execution with persistent credentials** and should not be run on a normal personal or enterprise workstation.

The core security concerns are fairly straightforward:

- Prompt injection through untrusted messages or content feeds.
- Credential theft or abuse.
- Malware hidden inside skills or extensions.
- Unsafe remote control over local tools and files.
- Persistence through saved memory, schedules, or configuration changes.
- Blurred boundaries between personal and organisational data.

This is why OpenClaw became both a breakthrough and a controversy. It made the agent real, but it also made the risks real.

## The Industry Reaction

OpenClaw did not remain a niche open-source curiosity. It triggered a broader response across the AI ecosystem.

### OpenAI

Public reporting in February 2026 said that Peter Steinberger, OpenClaw's creator, joined OpenAI and that Sam Altman said the project would continue as an open-source foundation supported by OpenAI. The important point for this module is not the corporate structure but the signal: OpenAI clearly viewed personal agents and agentic gateways as strategically important enough to hire the person most associated with the category.

### NVIDIA

NVIDIA's response was official and explicit. In its March 16, 2026 newsroom announcement, NVIDIA introduced **NemoClaw** for the OpenClaw community. The pitch was very revealing: keep the gateway idea, but add a stronger infrastructure layer beneath it, including privacy controls, policy-based guardrails, isolated runtime components, and support for both local and cloud models. In other words, NVIDIA accepted the gateway thesis and tried to harden it for more trustworthy deployment.

### China

The Chinese market moved quickly. Reuters reported in March 2026 that Baidu launched a family of agents tied to the OpenClaw wave, while Chinese authorities also warned about the security risks of poorly configured OpenClaw deployments. That combination is important: rapid adoption on one side, rapid concern on the other. The gateway idea was seen not just as a hobbyist trick, but as a serious new software category.

## Where Anthropic Fits

Anthropic did not position Claude as "an OpenClaw clone," and it would be sloppy to say that. But it is fair to say that Anthropic moved toward the same underlying user demand: people want agents that can do real work across tools, files, and surfaces, while giving them stronger guardrails.

Anthropic's answer is more managed and permissioned:

- **Claude Code** uses explicit permission prompts, restricted write scope, and configurable security settings.
- **Cowork** brings agentic capabilities into Claude Desktop, but Anthropic explicitly warns users to avoid sensitive files, use trusted MCPs, and treat the feature as a research preview with unique risks.
- **Dispatch** gives remote task assignment through Claude's own mobile and desktop surfaces rather than through an open multi-channel gateway.

So the contrast is not:

- OpenClaw = agentic.
- Claude = not agentic.

The real contrast is:

- **OpenClaw** pushes for maximum openness, channel reach, and user sovereignty.
- **Anthropic** pushes for more controlled, permissioned, productised agent surfaces.

That is a design and governance difference, not only a feature difference.

## Why This Matters

By this point in the module sequence, participants have seen:

- The rise of Claude Code as an agentic coding tool.
- The rise of Cowork as an agentic workbench.

Agentic gateways add a third idea:

the agent does not only need a work surface, it may also need a **communications surface**.

That communications layer is what turns a local runtime into something that feels reachable, continuous, and personal.

## Comparison Table

| OpenClaw functionality | Safety, security, and privacy issues | Claude Code / Claude Cowork functionality | How Claude addresses the issues and what the benefits are |
|---|---|---|---|
| Remote control through messaging apps such as Telegram, Discord, WhatsApp, and iMessage | Broad ingress surface; untrusted messages can carry prompt injection; auth and channel exposure become critical | Claude Code stays primarily inside terminal and IDE surfaces; Cowork uses Claude Desktop with Dispatch as a more product-controlled remote surface | Fewer open inbound channels reduce exposure; Anthropic keeps the remote path inside its own app surfaces, which is easier to govern and explain |
| Single gateway process connecting channels, sessions, models, hooks, and tools | A central gateway can become a high-value target; misconfiguration can expose many systems at once | Claude Code uses per-session local execution with explicit permissions; Cowork runs through Claude Desktop with narrower product boundaries | The blast radius is smaller and more legible; permission prompts and session boundaries make risky actions easier to review |
| Plugin and skill installation from a fast-moving ecosystem | Third-party skills may contain malicious code or unsafe behavior | Claude Code supports MCP and skills but documents trust expectations and permission controls; Cowork packages extensions more deliberately through plugins and customization | More managed packaging and explicit permissioning help reduce "install random code from the internet" risk, even though trust is still required |
| Persistent memory, schedules, and long-running task state | Persistence can preserve malicious instructions or risky approvals over time | Claude Code has project instructions, memory, and settings; Cowork has scheduled tasks and persistent project context | Anthropic explicitly frames these as governed features with permission prompts and safety guidance, which makes persistence more manageable for ordinary users |
| Local file and tool access from a remotely reachable agent | Exfiltration and destructive actions become possible if the agent is manipulated | Claude Code is read-only by default and restricts writes to the working tree; Cowork requires users to choose which local files to expose and warns against sensitive data | File access is narrower and more visible; Anthropic's safety model is designed around asking before high-risk actions rather than assuming full ambient authority |
| Open, model-agnostic "bring your own provider" architecture | Flexible routing can complicate data governance, auditing, and responsibility boundaries | Claude surfaces are more vertically integrated around Anthropic's product stack, even when external tools are connected | The trade-off is less flexibility but clearer accountability, more consistent UX, and stronger default guardrails |

## Key Takeaway

OpenClaw matters because it made the category visible. It showed that the next battle was not only about which model is smartest, but about who controls the gateway between people, agents, tools, and channels.

Anthropic's response matters because it shows a different bet: you can build powerful agents without making the gateway completely open. You can trade some openness for more permissioning, clearer boundaries, and stronger safety defaults.
