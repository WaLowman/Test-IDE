# TAXONOMY

**Bringing order to the various agentic constructs**

## Body

The purpose of this taxonomy is to stop everything from collapsing into one vague label such as "AI agents." The previous notes in this module showed that very different products are now being grouped together:

- Claude Code.
- Claude Cowork.
- OpenClaw.
- Cursor.
- LangGraph.
- Custom lead-generation workflows.

They are all agentic in some sense, but they are not doing the same job. This taxonomy gives us a way to distinguish the **surface**, **runtime**, and **role** of each system.

At the highest level, I use **Agentic Workspaces (AWS)** as the umbrella term. An agentic workspace is any environment in which an AI agent can help structure, execute, or coordinate real work rather than merely answer one-off prompts.

### 1. Agentic Workspaces (AWS)

This is the umbrella category. It includes any setup in which the agent is no longer just a chatbot but part of a working environment. The key idea is that the agent can operate with some combination of:

- Files.
- Tools.
- Memory.
- Instructions.
- Repeated procedures.
- User supervision.

So "workspace" is the broadest term. Everything else in the tree is a subtype.

### 2. Agentic Workbenches (AWB)

An **Agentic Workbench** is a broad work surface for non-trivial professional tasks. It is designed for operational work, not only code.

This is where **Claude Cowork** belongs. As the previous article argued, Cowork takes the local, tool-using, multi-step architecture of Claude Code and repackages it as a desktop workbench for broader knowledge work. The centre of gravity is not the repo or terminal. It is things like:

- Reports.
- Spreadsheets.
- Presentations.
- Research tasks.
- Recurring briefings.
- Connected office work.

So an agentic workbench is best understood as a **general-purpose operational surface**.

### 3. Agentic Coding Tools (ACT)

An **Agentic Coding Tool** is a tool whose natural home is the command line and whose primary orientation is code, repos, files, and execution.

This is where **Claude Code** belongs. In the earlier article, the key insight was that Claude Code began as a coding assistant but was "abused" into a more general-purpose operator. Even so, its native grammar is still that of:

- Terminal sessions.
- Git repos.
- Code changes.
- Structured files.
- Commands and scripts.

So even when Claude Code is used for general-purpose work, it remains an agentic coding tool because the **surface of use** is still the coding environment.

### 4. Agentic Coding Editors (ACE)

An **Agentic Coding Editor** is not just the agent and not just the IDE. It is the combination of an editing environment plus agentic assistance woven into the editing experience.

This is where tools like **Cursor**, **VS Code plus Copilot**, and **Google Antigravity** fit. These systems matter because they make the editor itself the working surface. In the Claude Code story, the IDE was the first place where prompting started to become operational. ACEs extend that idea by making the editor:

- A prompt surface.
- A file surface.
- A code surface.
- A review surface.
- An orchestration surface.

So the distinction is:

- **ACT** = the terminal-first agent.
- **ACE** = the editor-first environment.

### 5. Agentic Workflows (AWF)

An **Agentic Workflow** is a repeatable process that has been designed around one or more agents. It is not mainly a product category. It is a **process category**.

This is where your own systems belong, such as the lead-generation workflow described in the Claude Code note. Once prompts become repeatable, outputs become structured, and sub-agents or steps are coordinated, you no longer have just a tool session. You have a workflow.

Examples of agentic workflows include:

- Discovery -> ranking -> research -> outreach pipelines.
- Recurring content production systems.
- Project management assistants.
- Proposal-generation flows.

The important point is that a workflow can sit on top of several other categories. You might build one using Claude Code, run part of it in Cowork, and expose parts of it through a gateway.

### 6. Agentic Frameworks (AFW)

An **Agentic Framework** is the software infrastructure used to build or orchestrate agents programmatically.

This is where **LangChain**, **LangGraph**, **CrewAI**, and **Agent Zero** belong. These are not primarily end-user work surfaces. They are builder tools. They matter when you want to create systems such as:

- Routing logic.
- Agent graphs.
- Toolchains.
- Memory layers.
- Multi-agent orchestration.
- Evaluation and control logic.

So frameworks sit lower in the stack. They are the construction layer rather than the everyday user surface.

### 7. Agentic Gateways (AGW)

An **Agentic Gateway** is the communications layer that lets an agent be reached through external channels such as Telegram, Discord, WhatsApp, or iMessage.

This is where **OpenClaw** belongs. As the previous article argued, OpenClaw's importance lies in turning messaging apps into remote control surfaces for personal agents. That means the gateway is not just another interface. It is the layer that connects:

- The user.
- The messaging channel.
- The runtime.
- The tools.
- The credentials.
- The files and systems the agent can touch.

This makes gateways powerful, but also risky. They enlarge the surface area of the agent by making it remote, persistent, and reachable through multiple channels.

### Why These Distinctions Matter

Without this taxonomy, people make category mistakes such as:

- Calling Claude Cowork a chatbot.
- Treating OpenClaw as just another app.
- Treating Claude Code and Cursor as the same thing.
- Confusing a workflow with a framework.

The taxonomy helps because it separates:

- **What the user touches.**
- **What the agent runs inside.**
- **What kind of work is being done.**
- **Whether the system is a product, a process, or a builder layer.**

### The Main Practical Distinctions

If you need a simple way to remember the categories, use this:

- **Workbenches** are for broad operational work.
- **Coding tools** are for terminal-first agentic execution.
- **Coding editors** are for editor-first agentic work.
- **Workflows** are repeatable agent-driven processes.
- **Frameworks** are the builder infrastructure.
- **Gateways** are the remote communications layer.

### Key Takeaway

This taxonomy provides a mental map for the rest of the course. It explains why Claude Code, Claude Cowork, OpenClaw, Cursor, and LangGraph all belong in the same conversation, while also making clear that they occupy different layers of the agentic stack.
