# ABUSE OF CLAUDE CODE

**How a Coding Tool Became a General-Purpose Operating Layer for Knowledge Work**

Claude Code is the first concrete example of the broader shift described in the previous chapter. What began as a coding tool gradually became a more general-purpose local operator and, in doing so, helped open the path towards workbenches, workflows, and the wider agentic stack.

## From Prompting To Local Operation

My first experiments inside an IDE were not really about software engineering. In **February 2025**, I was using an IDE mainly as a better place to think and prompt. At the time, ChatGPT had a "Work With" feature on macOS that connected the app to an editor, and that changed my mental model. The IDE stopped being just a programmer's space and became a structured environment for working with AI.

That was the first shift: from **chat as a destination** to **the editor as a workbench**.

The second shift came when I started trying to move beyond prompting and into execution. Once MCP and related integrations started appearing, it became possible to connect Claude to external tools and data sources. But I was not really interested in sending work outward through a chain of services if I could avoid it. What I wanted was much simpler and much more powerful: I wanted the model to work **directly on my machine**, against my own files, in my own folders, with a clear record of what had happened.

By around **August 2025**, that became practical. I started using Claude Code through the editor, especially against documents in my Obsidian vault. At that point, Claude Code stopped feeling like a coding assistant and started feeling like a **local operator**. It could read, manipulate, review, and reorganise real working material. That is the moment the tool crossed the line from "help me write prompts" to "help me do the work."

From there, the pattern expanded naturally. I began using Cursor, the Claude Code extension, and Codex not only to draft prompts, but to improve prompts, execute them, and preserve the full history of a project inside a GitHub repo. This mattered because the repo became more than version control. It became:

- A memory of decisions.
- A record of iterations.
- A place where prompt chains turned into workflows.
- A bridge between human direction and agent execution.

Once that infrastructure was in place, the next move was obvious: stop treating each prompt as an isolated interaction and start designing **systems of work**. That led to orchestrators, sub-agents, structured outputs, and eventually full agentic workflows. In my case, that culminated in a lead-generation workflow for an expatriate management consulting agency. The important point is that this did not begin as "AI automation." It began as a practical attempt to make prompting more durable, more local, and more operational.

So the real story of Claude Code is not that a coding tool became popular with non-coders by accident. The deeper story is that once the tool could operate on local files, run commands, preserve context, and coordinate agents, it became useful for any knowledge worker willing to structure work as files, folders, prompts, and repeatable procedures.

## The Bigger Shift

This is why I describe the period as the "abuse" of Claude Code. People did not only use it for vibe coding or software tasks. They started using it for:

- Document operations.
- Research pipelines.
- Proposal drafting.
- Workflow execution.
- Repo-based project management.
- Agent orchestration across business tasks.

In other words, Claude Code escaped the narrow category of "coding assistant" and gave birth to a broader class of **agentic workbenches**.

## What Changed Conceptually

The sequence above shows a clear progression inside Claude Code:

1. **Better models** made Claude more reliable.
2. **Claude Code** gave Claude a serious local execution environment.
3. **Memory and Skills** made work more persistent and reusable.
4. **Wider organisational access** pushed Claude Code from power-user tool toward mainstream team workflow.

That is the storyline of this chapter. The important lesson is not simply "Claude got better." The lesson is that the centre of gravity shifted:

- From chat to agentic coding tool.
- From prompts to procedures.
- From isolated answers to persistent workflows.
- From single interactions to agentic systems.

## Key Takeaway

The practical implication is straightforward: if the work can be expressed as files, folders, prompts, instructions, and repeatable steps, Claude no longer has to remain a chatbot. It can become part of the operating layer of the work itself.
